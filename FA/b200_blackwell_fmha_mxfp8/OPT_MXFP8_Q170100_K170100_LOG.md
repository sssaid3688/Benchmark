## Iteration 1: TMEM/O experiments

### Attempt A: single O by aliasing O1 to O0

- Change tried: `PipelineO<1>`, `tOtO1 -> O0`, correction stage 1 reads/rescales `O0`, and S0/S1 loop ordering changed to advance S before the next single-O PV acquire.
- Result: no deadlock after S0/S1 reordering, but incorrect for multi-K shapes.
- Evidence:
  - `q=128,k=128`: `O max_diff=0.001587`, `passed_O=1`.
  - `q=128,k=512`: `O max_diff=4.687500`, `passed_O=0`.
  - `q=512,k=512`: `O max_diff=1.060547`, `passed_O=0`.
- Conclusion: in the current `ThreadShape<2,1,1>` design, `O0` and `O1` are long-lived accumulators for two Q halves across all K tiles. Simple aliasing overwrites the other half's accumulated O when `K > 128`.

### Attempt B: keep dual O and move SF after O1

- Change tried: keep `O0/O1` and move SF to columns after O1, making TMEM footprint exceed the original 512-column budget.
- Result: kernel illegal memory access on `q=512,k=512`.
- Conclusion: this path likely exceeds the supported TMEM allocation footprint for this kernel. Independent SF plus dual O does not fit the current 512-column budget.

### Current safe state

- Restored 512-column TMEM layout with dual O accumulators.
- Kept a validated loop ordering that advances `S0` before the first PV in the steady-state loop.
- Small-shape verification:
  - `q=512,k=512`: `O max_diff=0.006744`, `passed_O=1`.
  - `q=1024,k=1024`: `O max_diff=0.005432`, `passed_O=1`.
  - `q=4096,k=4096`: `O max_diff=0.003006`, `passed_O=1`.
- Big-shape performance:
  - Normal program timing printed `778.187 TFLOPS/s` for `q=170100,k=170100` with `--iterations=1`; this is not the acceptance metric.
  - ncu CSV: `/home/ubuntu/workspace/oyhj/bigshape/Benchmark/ncu_iter1_safe_q170k170.csv`.
  - Main kernel ncu durations: `1.235790304s` and `1.235052512s` across the two SpeedOfLight passes.
  - Manual TFLOPS from ncu duration: `479.51` and `479.79 TFLOPS`.
  - ncu bottleneck indicators: SM throughput about `59.4%`, DRAM throughput about `0.09%`, memory throughput about `21.9%`, `No Eligible=52.19%`, registers/thread `128`.

## Reference implementation analysis

User-provided reference:

`/home/ubuntu/workspace/chenyun/b200_benchmark_test/src/operators/op6_mxfp8_fmha/custom/sm100_fmha_fwd_mainloop_mxfp8_n128_pvmx.hpp`

Key finding:

- The reference single-O design is not a direct `O1 -> O0` alias in the current two-Q-half mainloop.
- It changes to `ThreadShape<1,1,1>` / single-stage N128:
  - one M=128 subtile per CTA,
  - `O1 = O0` because stage 1 is never live,
  - `PipelineO<1>`,
  - `PipelineS<2>` for double-buffered S,
  - SF region fits inside the 512-column TMEM budget.
- Therefore the correct next optimization is to add an N128/single-stage specialization and route the target shape to it, including matching mainloop, load, kernel schedule, and epilogue changes.

## Next step

- After porting, rerun small-shape verify and big-shape ncu performance.

## Iteration 2: Single-O correction critical path

### Latest baseline

- User report: `/home/ubuntu/workspace/oyhj/handTest/Benchmark/report_full_900_1.ncu-rep`.
- Main-kernel duration: `1.038111s`.
- Manual performance: `592568524800000 / 1.038111 = 570.82 TFLOPS`.
- Current path is M=128 single-stage with one O accumulator, double-buffered S,
  split-N softmax, and `StageCountKV=4`.

### Retained change: release softmax stats before waiting for O

- Bottleneck analysis:
  - Correction loaded row stats from S/V and computed the rescale factor, but
    held `pipeline_s0_c` while waiting for and rescaling O.
  - After row stats are in registers, correction no longer needs the S/V
    buffer. Holding the signal unnecessarily blocks the next softmax/P/SFP
    tile.
- Change:
  - Release `pipeline_s0_c` immediately after loading stats and computing
    `scale`, before `pipeline_o.consumer_wait`.
  - Acquire the tail epilogue pipeline before waiting for final O.
  - TMEM layout, single-O ownership, and numerical operation order are
    unchanged.
- Correctness:
  - `q=k=512,h=40`: `O max_diff=0.006744`.
  - `q=k=1024,h=40`: `O max_diff=0.005432`.
  - `q=k=4096,h=40`: `O max_diff=0.003006`.
- NCU duration: `1.029857120s`.
- Manual performance: `575.39 TFLOPS`.
- Improvement over latest baseline: about `0.80%`.
- Rebuilt after reverting rejected KV-stage experiments and remeasured:
  `1.029888736s`, `575.37 TFLOPS`.

### Rejected experiments

- Persistent scheduler:
  - NCU duration: `1.059961568s`.
  - Slower than retained individual scheduler by about `2.9%`.
- `StageCountKV=5`:
  - Correctness passed.
  - NCU duration: `1.039202336s`.
  - Slower than retained version by about `0.91%`.
- `StageCountKV=3`:
  - Correctness passed.
  - NCU duration: `1.088795168s`.
  - Slower than retained version by about `5.72%`.
- `NumRegsSoftmax=160`:
  - Correctness passed.
  - NCU duration: `1.031736992s`.
  - Slightly slower than retained `NumRegsSoftmax=176`.
- Conclusion: keep IndividualTileScheduler and default `StageCountKV=4`.

### Next bottleneck

- The correction release optimization produced only a small gain, so the
  largest remaining cost is likely inside split-N softmax/P/SFP generation or
  the mandatory single-O PV/rescale chain.
- Next iteration should profile and optimize split-N NamedBarrier/reduction and
  P/SFP generation before changing the single-O pipeline again.

## Iteration 3: Investigate high warp-wait via conditional O rescale

- Hypothesis:
  - The reported high warp-wait could be caused by MMA waiting for correction
    to release the single O accumulator.
  - When a row's running maximum is unchanged, its correction scale is exactly
    one, so rewriting O appears avoidable.
- Experiment:
  - A per-row conditional around `correction_rescale` deadlocked because the
    TMEM copy atom requires uniform warp-group participation.
  - Replaced it with a uniform `any row needs rescale` decision using shared
    memory and two 128-thread NamedBarriers.
- Correctness for the uniform version:
  - `q=k=512`: `O max_diff=0.006744`.
  - `q=k=1024`: `O max_diff=0.005432`.
  - `q=k=4096`: `O max_diff=0.003006`.
- NCU result:
  - Duration: `1.157305440s`.
  - Manual performance: `512.04 TFLOPS`.
  - This is about 12.4% slower than the retained `1.029888736s` version.
- A lower-cost warp-ballot reduction exposed an invalid assumption about the
  correction thread layout and failed with `misaligned address`.
- Conclusion:
  - High warp-wait is a real latency symptom, but this experiment does not
    establish unconditional O rescale as a profitable root cause.
  - The extra synchronization required to make the skip uniform costs more than
    the avoided rescale work.
  - Conditional O-rescale experiments were reverted. Continue with the retained
    early stats release and investigate the P-ready/split-N softmax wait chain.

## Iteration 4: Warp-wait and long-scoreboard localization

- Lightweight NCU stall metrics on the retained version:
  - Total average warp latency: 3,084,103.89 cycles.
  - Long scoreboard: 1,384,947.56 inst/warp (44.9% of warp latency).
  - Wait: 425,215.98 inst/warp (13.8%).
  - MIO throttle: 259,477.80 inst/warp (8.4%).
  - Barrier: 185,140.34 inst/warp (6.0%).
  - Short scoreboard: 138,751.54 inst/warp (4.5%).
  - Membar: 0.
- Conclusion: warp wait is a meaningful bottleneck symptom, but long scoreboard is
  the largest measured stall source and is about 3.25x larger than wait.
- PM sampling completed but this Nsight Compute/B200 combination returned only
  elapsed cycles and duration, not per-PC warp-state attribution. Full
  SourceCounters replay was stopped because it exceeded the practical runtime.

### Rejected experiment: defer wait V(k) until after PV(k-1)

- Rationale: V(k) is used in the next iteration, so moving its wait after the
  current PV could overlap V TMA latency with PV execution.
- Correctness passed:
  - q=k=512: O max_diff=0.006744.
  - q=k=1024: O max_diff=0.005432.
- NCU duration: 1.047607648s, or 565.62 TFLOPS.
- This is about 1.72% slower than retained 1.029888736s.
- Conclusion: the existing early V wait participates in a better pipeline/release
  cadence; mechanically moving it does not reduce the effective critical path.
  The experiment was reverted.

## Iteration 5: TMA dependency and split-N synchronization

### TMA/cache bottleneck analysis

- Target-shape small-group NCU metrics:
  - Duration: `1.030149376s`.
  - DRAM read throughput: about `6.48 GB/s`.
  - TMA global-load bytes: `2,388,290,826,240`.
  - TMA requests: `285,628,680`.
  - L2 sectors hit: `61,360,668,014`.
  - L2 sectors miss: `579,335,538`.
  - Approximate L2 hit rate: `99.06%`.
  - Long scoreboard remained `1,384,976.58 inst/warp`.
- Conclusion:
  - Long scoreboard is not caused by HBM bandwidth or DRAM misses.
  - The likely issue is TMA request/completion dependency and request
    granularity. Each K/V tile also issues a separate small scale-factor TMA.

### Rejected synchronization experiments

- Group 1 non-blocking arrive at `B_PDONE`:
  - Correctness passed for 512/1024/4096.
  - Barrier stall dropped from `185,140` to `142,634 inst/warp`.
  - Duration regressed to `1.037125888s`; reverted.
- Group 0 non-blocking arrive at `B_SREADY`:
  - Correctness passed for 512/1024/4096.
  - Duration regressed to `1.042276320s`; reverted.
- Conclusion: keeping the two softmax groups lockstepped improves overall
  resource scheduling even when it increases the barrier metric.

### Retained: delay correction producer acquire past row-sum

- Moved the depth-1 correction `producer_acquire` from before row-sum to after
  row-sum so correction consumption can overlap independent reduction work.
- Correctness:
  - q=k=512: O max_diff=0.006744.
  - q=k=1024: O max_diff=0.005432.
  - q=k=4096: O max_diff=0.003006.
- Target-shape NCU durations:
  - `1.029804224s`
  - `1.029783904s`
  - `1.029773920s`
- Median: `1.029783904s`, or `575.42 TFLOPS`.
- Improvement is only about `0.01%`, but is repeatable and preserves the
  pipeline operation counts and numerical ordering.

### Retained: prefetch current V while K is in flight

- Ported the V prefetch pattern used by the repository's Blackwell MLA load
  collective into the N128 steady-state K/V load loop.
- V prefetch does not change pipeline transaction bytes, consumption order, or
  numerical operations.
- Correctness passed for q=k=512/1024/4096 with unchanged max differences.
- Long scoreboard decreased from about `1,384,976` to `1,380,957 inst/warp`.
- TMA requests increased from `285,628,680` to `356,225,160`.
- Target-shape durations:
  - `1.029658304s`
  - `1.029699808s`
  - `1.029653056s`
- Median: `1.029658304s`, or `575.49 TFLOPS`.

### Rejected prefetch variants

- Prefetch both current V and next K:
  - Long scoreboard decreased to `1,375,100 inst/warp`.
  - TMA requests increased to `426,821,640`.
  - Duration regressed to `1.031641024s`; next-K prefetch was removed.
- Prefetch SFV instead of V:
  - Long scoreboard decreased to `1,363,492 inst/warp`.
  - SFV's fragmented layout expanded prefetch traffic to `2,544,716,040`
    TMA requests.
  - Duration regressed to `1.033534496s`; reverted.
- Conclusion: SF request granularity contributes to the scoreboard latency, but
  issuing additional SF prefetches costs much more than the latency reduction.
  A future higher-impact change must merge or replace the independent SF TMA
  transaction rather than prefetch it.

## Iteration 11: P-quantization fusion experiment

- Analyzed moving P quantization into the fused softmax `exp2 + amax` loop.
  Full immediate quantization is not possible because each 32-element MXFP8
  block must finish its amax reduction before its scale is known.
- Tested a block-local schedule:
  `32x exp2+amax -> scale -> quantize`, repeated for the second block.
- Correctness remained unchanged at q=k=1024:
  `O max_diff=0.005432`.
- Three target-shape NCU durations were `952.862336ms`, `951.971104ms`, and
  `951.032928ms`; median `951.971104ms`.
- Compared with the retained 32x-TMEM-load baseline median `951.076736ms`,
  block-local quantization regressed about `0.09%`; reverted.
- Also tested fusing row-sum accumulation into P generation. This extended four
  dependent add chains on the P-ready critical path and regressed duration to
  `969.033600ms`; reverted.
- `report620` source correlation maps the high-frequency ordinary shared load
  to split-N `row_max` exchange. The instruction itself reports `N-way=1` and
  exactly one ideal/actual wavefront per request, despite the aggregate bank
  conflict counter. Therefore padding or swizzling this exchange cannot remove
  the aggregate count and would not be an evidence-based optimization.
- Conclusion: keep exp2+amax together and quantization as the following
  independent pass. The large aggregate shared-load conflict count is not an
  intra-warp layout conflict; the measurable bottleneck remains softmax TMEM
  dependency latency and the single-O pipeline.
- Restored retained version verification:
  - q=k=512/1024/4096 correctness passed with O max_diff
    `0.006744/0.005432/0.003006`.
  - Three target-shape durations: `951.514144ms`, `951.408384ms`,
    `952.476128ms`; median `951.514144ms`, or `622.76 TFLOPS`.
